# Ralph Progress Log

## Codebase Patterns

### How to query episode_summaries.jsonl
NEVER read the file directly (2.3MB). Always use `python3 -c` via Bash:
```python
import json
lines = open('artifacts/episode_summaries.jsonl').readlines()
eps = [json.loads(l) for l in lines]
# then aggregate
```

### Episode summary structure
Keys: altitude_at_apogee_m, altitude_at_burnout_m, altitude_ratio_to_target,
curriculum_target_altitude_m, episode, fuel_used_fraction, gimbal_variance,
max_altitude_m, max_g_load, max_q_dyn, mean_throttle, reward_components,
success, t, termination_reason, throttle_variance, time_to_apogee_s

reward_components keys: coasting_pen, fuel_pen, g_pen, omega_pen, progress,
q_pen, smooth_pen, survival, terminal_bonus, terminal_penalty, tilt_pen

### How reward is computed in env.py
- step(): computes per-step reward components, accumulates into self.reward_sums
- terminal reward: added at episode end (success +100, failure -(50+25*err))
- info_dict() populates episode summary via train.py callback
- _progress_potential(): phi = clip(alt/target, 0, 2) — pure altitude shaping

### How telemetry fields get into episode_summaries.jsonl
1. env.py info_dict() returns a dict with all episode stats
2. train.py callback reads info["reward_components_cumulative"] at episode end
3. Writes a JSON line to artifacts/episode_summaries.jsonl with flattened fields
4. To ADD a new field: add it to info_dict() return dict in env.py, then ensure train.py callback writes it

### Curriculum target
- Set via env.set_curriculum(target_altitude_m) in train.py CurriculumManager
- Stored as self.curriculum_target_altitude_m (float, meters)
- Only changes when success_threshold met over rolling window
- phi = alt / curriculum_target_altitude_m (not mission.target_altitude_m)

## 2026-03-25 - S3: Reduce terminal penalty coefficient to free up reward gradient

### What was done
- Changed `fail_pen = 50.0 + 25.0 * terminal_penalty` to `fail_pen = 25.0 + 10.0 * terminal_penalty` in env.py
- Added comment explaining the reasoning (was 67% of gradient, now ~48%)
- No other reward terms touched; success bonus (+100) and stall_pen (25.0) unchanged

### Reasoning
- Run 12 data: terminal_penalty avg -58.4, non-terminal sum +14.1 → penalty was 80% of negative signal
- At avg err_cost=0.336: old fail_pen = 50+25*0.336 = 58.4; new = 25+10*0.336 = 28.4
- Ratio of terminal-to-shaping drops from ~4:1 to ~2:1, allowing progress/q signals to matter more
- Specific: at 86km miss (err=0.14 alt fraction, stage="high", total err_cost ~0.34):
  old penalty = -(50+25*0.34) = -58.5; new = -(25+10*0.34) = -28.4 (52% reduction)
- Falsification: if avg altitude drops (policy stops caring about reaching target) or success stays flat

### Risks noted
- Both S3 and S4 act in the same direction (amplify shaping signals) — they are being batched into Run 13
  together since the PRD says S4 should only be added after S3 "validated or user decides to batch"
- The stall_pen (25.0 extra for very early failures) is unchanged — maintains a strong floor penalty

### Confidence
- Implementation correct: 10/10
- Hypothesis valid: 9/10
